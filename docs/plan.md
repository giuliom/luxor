## Plan: Production Rust Backend Template

Transform the compact Axum demo into a runnable reference backend: centralized configuration and observability, PostgreSQL persistence via SQLx migrations, Redis cache/queue interfaces, JWT access/refresh authentication, provider-agnostic OAuth2 boundaries, standardized errors, focused tests, and service-backed GitHub Actions. Retain the simple static frontend but make it exercise health, authentication, protected routes, caching, and enqueueing.

**Steps**
1. Establish the application foundation. Add modules for `config`, `error`, `state`, `observability`, `db`, `cache`, `auth`, `queue`, `handlers`, `models`, and `services`; revise `src/lib.rs` exports and replace `server::app()` with an async application constructor that accepts initialized `AppState`. Move listener/config initialization into `src/main.rs`, including graceful shutdown. *Blocks later route and dependency-injection work.*
2. Add environment-backed configuration. Define a validated `Config` with `APP_HOST`, `APP_PORT`, `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`, access/refresh lifetimes, `OTEL_EXPORTER_OTLP_ENDPOINT` (optional), `SENTRY_DSN` (optional), and OAuth provider metadata (optional). Add `.env.example`, update `.gitignore` for `.env`, and document secret handling. Startup must fail clearly for required production credentials while allowing a documented development configuration.
3. Add observability and consistent failures. Initialize `tracing-subscriber` with compact development output and JSON production output controlled by `RUST_LOG`/environment. Add request IDs and `tower-http` request spans, propagate the request ID in responses, map errors through an `AppError`/`ApiErrorResponse` contract, and initialize Sentry only when its DSN exists. Add OpenTelemetry tracing only when an OTLP endpoint is configured, ensuring logs and traces share request context and shutdown flushes telemetry.
4. Provide local infrastructure and SQLx migrations. Add a Compose file for PostgreSQL and Redis using health checks, persistent local volumes, and ports/config that match `.env.example`. Add `migrations/` managed by SQLx CLI: `users`, refresh-token/session records, and a sample domain record or audit table needed by the demo. Add a SQLx pool initializer with connection settings, `sqlx::migrate!()` startup migration support for development only (document that production deploys migrations explicitly), repository functions using compile-time-safe/query-checked SQL where practical, and indexes/constraints for token lookup/revocation.
5. Implement Redis-backed cache and queue boundaries. Use async Redis connection management in `AppState`. Implement a typed cache interface with namespaced keys, TTLs, invalidation, and graceful cache-miss behavior. Implement a typed queue trait using a documented Redis list/stream strategy, structured payload serialization, retry metadata, and enqueue-only API semantics; no worker executable is included by the user decision. Add a `JobKind` sample such as `SendEmail`/`AuditEvent` without choosing an email provider.
6. Implement authentication and OAuth2 extension points. Add password hashing and secure user registration/login endpoints; issue short-lived signed JWT access tokens plus opaque, hashed, rotating refresh tokens stored in PostgreSQL. Add `Authorization: Bearer` extraction middleware and a protected `GET /api/me` endpoint. Add logout/revocation and refresh rotation. Define provider-agnostic `OAuthProvider`/`OAuthIdentity` interfaces plus optional configuration validation, callback-state handling primitives, and clear docs for adding a provider; do not include a provider-specific login flow. Use only HTTP-only secure cookie handling for any browser-held refresh credential, keeping access tokens in frontend memory.
7. Reorganize the HTTP surface and preserve basic demo routes. Split health, auth, users, cache, and jobs into handlers, group routes under `/api`, apply CORS from configuration, body limits, tracing, and request-ID middleware. Retain/recast `GET /api/health`, `GET /api/hello`, and `GET /api/time` as lightweight checks. Add a protected cache demonstration and a protected enqueue endpoint, with JSON request/response DTOs and standardized error responses.
8. Update the static demo frontend. Keep static asset serving but revise `public/index.html`, `public/script.js`, and styles to provide registration/login/logout, visible authenticated-user state, protected-profile fetch, cache interaction, and queue-enqueue feedback. It must use same-origin fetches, send the refresh cookie only as required, retain access tokens only in JavaScript memory, and show server/API errors without leaking secrets. *Depends on steps 6-7.*
9. Expand automated testing. Preserve the existing `tower::ServiceExt` router tests but make the app injectable with test state. Add unit tests for config validation, JWT verification/expiry, token hashing/rotation, API error serialization, cache key/TTL behavior, and queue serialization. Add integration tests for migrations, repositories, login/refresh/logout/protected endpoint behavior, and Redis cache/queue behavior, gated by documented `DATABASE_URL`/`REDIS_URL` test services. Use isolated database setup/cleanup or per-test schema/database to prevent cross-test contamination.
10. Upgrade quality CI and documentation. Update `.github/workflows/CI.yml` to run `cargo fmt --check`, `cargo clippy -- -D warnings`, build, unit tests, and integration tests against GitHub Actions PostgreSQL and Redis services. Run SQLx migrations before database tests and configure only test-safe workflow environment values. Rewrite `README.md` with prerequisites (Rust, Docker Compose, `sqlx-cli`), setup commands, migration create/run/revert workflow, local services, route/auth flow, test commands, environment variables, observability/Sentry setup, OAuth provider integration steps, queue-worker contract, and a production secrets/deployment checklist. Do not add a deployment or image-publishing workflow.

**Decisions**
- Primary auth is JWT access tokens with rotating refresh tokens; browser refresh credentials are HTTP-only cookies and the access token is in-memory only.
- OAuth2 is intentionally provider-agnostic: abstractions, config, and extension documentation are included; no GitHub/Google-specific endpoints or credentials are committed.
- Redis queue is an abstraction plus enqueue API, not a worker executable. The message contract is designed to support a later separate worker crate/binary.
- CI covers quality and integration tests only. Docker publishing and deployment workflows are deliberately excluded.
- Email delivery remains provider-agnostic and deferred. A typed `SendEmail` job contract may exist, but no email provider SDK, credentials, or sending implementation is added.
- Required production secrets are environment variables; local secrets live only in ignored `.env`. Managed-secret provider wiring is deployment-specific and documented rather than hard-coded.
